# Phase 4: Document the contract and close the verification loop

## Objective

Make the implementation and user-facing documentation agree, then run the complete repository gates for issues #1–#5.

## Ordered tasks

### 1. Update README behavior and API documentation

Update `README.md` to include:

- `createAnalyticsReporter()` usage for a library/provider runner, including explicit `cwd`, `runsDir`, source, and lifecycle calls.
- The existing `pi install` path and the fact that `extensions/analytics.ts` is the Pi adapter.
- Request timeout as an overall deadline with abort semantics.
- Deterministic capped exponential retry policy (100/200/400 ms, capped at 2,000 ms), retryable statuses (408, 429, 5xx), and no retry for non-retryable responses.
- Queue overflow cap, oldest-drop behavior, immediate first diagnostic, rate-limited/aggregated subsequent diagnostics, and `stats()` metrics.
- Backstop at-least-once semantics: watermarks advance after successful contiguous delivery, failed/timed-out records replay after restart, atomic sidecars, and duplicates are possible after a crash.
- Explicitly unchanged privacy boundaries: only conductor telemetry, no role-session transcript upload, no secret logging.

**Acceptance:** Every delivery claim in README matches `src/queue.ts`, `src/watermark.ts`, `src/reporter.ts`, and adapter behavior; no “exponential” or outage-recovery wording remains inaccurate.

### 2. Add/update API and integration tests

- Ensure `tests/reporter.test.ts` validates the README's sample lifecycle and source labels.
- Ensure `tests/extension.test.ts` covers UI warning mapping for overflow without a conductor record context and confirms no duplicate subscriptions.
- Add a regression fixture or focused test using representative persisted conductor records, preserving unknown fields through the envelope.
- Keep tests deterministic and isolated from the developer's real HOME/config.

### 3. Run focused and full checks

Run in this order and record failures by root cause rather than weakening assertions:

```bash
pnpm vitest run tests/queue.test.ts tests/watermark.test.ts tests/reporter.test.ts tests/extension.test.ts
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

If a test relies on timers, verify the process exits cleanly after the command. Inspect generated `dist/` declarations to confirm `createAnalyticsReporter` and its types are published.

### 4. Final compatibility and stop review

- Review `git diff --check` and `git status --short`.
- Confirm `package.json` exports the built reporter API through the existing `dist/index.js` entrypoint and no direct pi-conductor dependency was added.
- Confirm low-level exports remain available or that any migration note is in README.
- Stop before release/merge if any acceptance criterion is only manually inferred, if a timer remains open, or if a failed backfill can advance a sidecar.

## Dependencies

Phases 1–3 complete, including focused tests and a stable public contract.

## Rollback/stop conditions

- Documentation must not be marked complete while implementation and README disagree.
- If full gates fail, isolate and fix the failing phase; do not reclassify a timeout, watermark, or diagnostic failure as flaky without evidence.
- If generated artifacts are tracked unexpectedly, restore only generated-file policy consistent with the existing repository; do not hide source/type failures by deleting declarations.

## Files likely touched

- `README.md`
- `tests/reporter.test.ts`
- `tests/extension.test.ts`
- `tests/envelope.test.ts` or a new fixture test
- `package.json` only if export metadata needs adjustment
- `dist/*` generated by `pnpm build` if repository policy tracks build output
